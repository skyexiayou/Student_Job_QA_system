# 使用说明

本文档基于项目最终版本整理，面向项目成员、演示人员和课程答辩使用。阅读后应能说明系统整体架构、功能模块、主要文件职责、数据流转过程、配置方式和演示路线。

## 1. 系统简介

大学生求职岗位知识问答系统是一个面向实习、校招、简历、面试和岗位选择场景的 RAG 智能问答系统。

系统允许用户上传求职相关资料，例如岗位 JD、企业招聘资料、面试题、简历指导文档、技术题库和行业信息。后端会对文档进行解析、清洗、分块、向量化和索引构建。用户提问时，系统先从知识库中检索相关内容，再结合历史对话和检索片段调用大模型生成回答，并在回答中展示参考来源。

项目不是单纯聊天页面，而是包含用户体系、知识库管理、岗位助手、模拟面试、知识图谱、语音输入、历史记录和目标管理的完整应用。

## 2. 用户与使用场景

目标用户是正在准备实习、校招和课程项目答辩的大学生。

典型场景：

- 想了解前端、后端、算法、测试、产品等岗位的能力要求。
- 想把课程设计、竞赛项目或实习经历整理成简历亮点。
- 想根据岗位 JD 准备技能清单、学习路线和项目介绍。
- 想进行模拟面试，让 AI 面试官点评并追问。
- 想上传企业资料或面试资料，快速问答和定位重点。
- 想在生产实习答辩中展示完整 AI 应用系统。

## 3. 系统整体架构

系统采用前后端一体部署方式。FastAPI 后端负责接口、静态资源服务和业务编排；前端是 Vue 3 单页应用，浏览器直接访问后端根路径即可进入系统。

```text
浏览器用户
  |
  | 页面访问、登录注册、上传文档、发送问题、录音输入
  v
frontend/index.html
  |
  | fetch HTTP 请求 / WebSocket 语音连接
  v
FastAPI 后端 src/rag_job_qa/api.py
  |
  +-- user_store.py        用户、令牌、头像、目标管理，连接 MySQL
  +-- knowledge_base.py    知识库导入、分类、检索、文档管理、图谱兜底
  +-- vector_store.py      Embedding、FAISS/numpy 检索、索引持久化
  +-- rag_service.py       RAG 主流程、提示词构造、大模型调用、来源追加
  +-- llm_client.py        大模型普通和流式接口调用
  +-- conversation.py      多轮会话和历史记录
  +-- neo4j_store.py       可选 Neo4j 知识图谱
  +-- speech_service.py    FunASR 语音识别
  +-- cache.py             问答缓存
  +-- models.py            数据模型
```

文档入库链路：

```text
上传 PDF/TXT/Markdown/DOCX/XLSX
  -> text_processing.py 解析文本
  -> clean_text 清洗文本
  -> split_text 分块
  -> knowledge_base.py 判断岗位分类并导入
  -> vector_store.py 生成向量
  -> data/index/ 保存 chunks、vectors、metadata
  -> neo4j_store.py 可选写入 Neo4j 图谱
```

问答生成链路：

```text
用户提问
  -> api.py 接收 /api/chat/stream
  -> rag_service.py 检查问题、会话和缓存
  -> knowledge_base.py 检索相关知识片段
  -> rag_service.py 构造系统提示词和用户提示词
  -> llm_client.py 调用大模型流式生成
  -> rag_service.py 追加参考来源
  -> conversation.py 保存历史记录
  -> 前端逐字显示回答
```

## 4. 前端页面与功能模块

当前前端集中在 `frontend/index.html`，使用 Vue 3 单页应用实现。页面通过 hash 路由切换：

| 页面 | 路由 | 主要功能 |
| --- | --- | --- |
| 首页 | `#home` | 系统入口、热门岗位、模拟面试入口、知识文档入口、求职目标倒计时 |
| 岗位助手 | `#jobs` | 展示岗位方向，按岗位分类关联文档，围绕岗位进行问答 |
| 知识图谱 | `#graph` | 展示岗位、文档、技能等节点和关系，支持搜索和分类筛选 |
| 知识问答 | `#chat` | RAG 多轮问答，支持流式回答、语音输入、来源展示和朗读 |
| 模拟面试 | `#interview` | AI 面试官追问，输出“面试官回应”和“简短分析” |
| 知识文档 | `#docs` | 上传、查看、筛选、预览、单个删除和批量删除知识文档 |
| 学习记录 | `#history` | 查看知识问答、模拟面试、岗位助手历史，支持重命名和删除 |

### 4.1 登录注册与个人中心

系统打开后，如果本地没有 token，会进入登录/注册界面。

支持功能：

- 用户注册。
- 用户名、邮箱或手机号登录。
- 退出登录。
- 修改昵称、邮箱、手机号。
- 上传头像。
- 修改密码。

相关接口：

```text
POST /api/auth/register
POST /api/auth/login
GET  /api/auth/me
POST /api/auth/logout
PUT  /api/auth/me
POST /api/auth/avatar
PUT  /api/auth/password
```

实现文件：

- `api.py`：定义认证接口。
- `user_store.py`：连接 MySQL，维护用户表、令牌表和目标表。
- `frontend/index.html`：保存 token 到 localStorage，并在请求时携带 Authorization Header。

答辩要点：

- 用户密码使用 PBKDF2 加盐哈希保存，不保存明文密码。
- 登录成功后生成 token。
- 后端通过 token 判断用户身份，实现用户数据隔离。

### 4.3 知识问答

知识问答是系统核心。用户输入问题后，系统会基于知识库检索结果生成回答。

支持功能：

- 多轮问答。
- 流式回答。
- 参考来源展示。
- 可设置召回片段数量。
- 附件上传并入库。
- 语音输入。
- 回答朗读。
- 回答复制。
- 停止生成。
- 历史会话保存。

相关接口：

```text
POST /api/chat
POST /api/chat/stream
GET  /api/sessions
GET  /api/history/{session_id}
POST /api/session/new
POST /api/session/{session_id}/clear
POST /api/session/{session_id}/rename
DELETE /api/sessions
```

实现文件：

- `rag_service.py`：RAG 编排、场景提示词、来源追加、缓存和会话写入。
- `knowledge_base.py`：检索知识片段。
- `vector_store.py`：向量相似度搜索。
- `llm_client.py`：调用大模型。
- `conversation.py`：保存历史对话。
- `cache.py`：缓存重复问题。

答辩要点：

- 系统不是直接问大模型，而是先查知识库。
- RAG 可以减少模型凭空回答的问题。
- 回答末尾有参考来源，便于追溯。
- 流式输出使用 `application/x-ndjson`，前端逐行读取事件。

### 4.5 岗位助手

岗位助手围绕具体岗位方向进行问答。前端内置多个岗位方向，后端使用岗位分类过滤知识片段。

后端主要岗位分类包括：

```text
backend     后端开发
frontend    前端开发
client      客户端开发
embedded    嵌入式底层开发
ai          AI 算法
bigdata     大数据开发
cloud       云计算/云原生
security    网络安全
testing     测试体系
game        游戏开发
ic          芯片 IC 开发
nondev      技术非开发岗
```

使用方式：

1. 进入岗位助手页面。
2. 选择一个岗位方向。
3. 查看该岗位相关文档。
4. 输入问题，例如“这个岗位需要准备哪些项目？”。
5. 系统带着岗位分类进行检索和回答。

实现方式：

- 前端发送 `session_type: "job"`。
- 前端发送岗位 `category`。
- `knowledge_base.py` 根据分类过滤召回结果。
- `rag_service.py` 为岗位助手加入岗位建议、能力要求、学习路径等场景提示。

### 4.6 模拟面试

模拟面试页面让 AI 扮演面试官。

支持功能：

- 自动生成开场。
- 用户输入自我介绍、项目经历或问题回答。
- AI 输出“面试官回应”。
- AI 输出“简短分析”。
- AI 继续提出一个具体追问。
- 支持语音输入。
- 支持停止生成。
- 面试会话进入历史记录。

实现方式：

- 前端发送 `session_type: "interview"`。
- 前端将候选人回答包装成面试官提示词。
- `rag_service.py` 使用面试场景提示。
- 回答仍会结合知识库检索结果。

演示建议：

可以先让同学说一段 1 分钟自我介绍，再展示 AI 如何追问项目细节、技术难点或岗位匹配度。

## 6. 基本使用流程

1. 启动 MySQL，并创建 `rag_job_qa` 数据库。
2. 配置 `.env` 或准备 API Key CSV。
3. 执行 `python run.py` 启动后端和前端页面。
4. 浏览器打开 `http://127.0.0.1:8000`。
5. 注册或登录账号。
6. 进入知识文档页面，上传求职资料。
7. 等待系统完成解析、分块、向量化和索引重建。
8. 进入知识问答页面提出问题。
9. 查看流式回答和参考来源。
10. 进入岗位助手，选择岗位后追问岗位准备建议。
11. 进入模拟面试，进行面试练习。
12. 进入知识图谱，查看文档、岗位和技能关系。
13. 进入学习记录，复盘历史会话。
14. 在首页添加求职目标，查看倒计时。

## 8. 答辩推荐讲解顺序

建议答辩时按照“问题背景 -> 系统架构 -> 核心流程 -> 功能演示 -> 技术亮点 -> 总结”的顺序讲。

### 8.1 问题背景

大学生求职资料来源分散，岗位 JD、面试题、简历经验和行业信息往往分布在不同文件中。普通搜索无法结合个人问题和已上传资料生成结构化建议。本系统通过 RAG 把本地知识库和大模型结合起来，提高求职资料利用效率。

### 8.2 系统架构

重点讲：

- Vue 单页前端。
- FastAPI 后端。
- MySQL 用户体系。
- 本地向量知识库。
- OpenAI 兼容大模型接口。
- 可选 Neo4j 知识图谱。
- 可选 FunASR 语音识别。

### 8.3 核心流程

重点讲两条链路：

- 文档上传后如何解析、分块、向量化和索引。
- 用户提问后如何检索、构造提示词、调用大模型和展示来源。

### 8.4 现场演示流程

1. 登录系统。
2. 展示首页和目标管理。
3. 上传一份岗位 JD 或面试资料。
4. 在知识问答中提问。
5. 展示参考来源。
6. 切换岗位助手，按岗位继续追问。
7. 切换模拟面试，让 AI 追问。
8. 展示知识图谱。
9. 展示学习记录。
10. 总结技术亮点。

### 8.5 技术亮点

可以强调：

- RAG 检索增强生成。
- NDJSON 流式回答。
- 支持多格式文档解析。
- 向量检索支持 FAISS，也支持 numpy 降级。
- Hugging Face 模型不可用时可降级到 hashing embedding。
- Neo4j 不可用时可降级到本地图谱。
- MySQL 用户体系和 token 登录。
- 语音输入和回答朗读提升交互体验。

## 9. 常见问题

### 9.1 没有 API Key 能不能运行？

页面、登录、知识库部分可以检查，但大模型回答会受影响。答辩前应确认 API Key 可用。

### 9.2 没有 Neo4j 能不能运行？

可以。Neo4j 是增强模块，不是必需模块。没有 Neo4j 时，系统仍会使用本地知识库生成基础图谱。

### 9.3 Hugging Face 模型没有下载怎么办？

系统会降级到本地 hashing embedding，可以保证演示可运行。检索效果可能不如正式 Embedding 模型。

### 9.4 上传文档后为什么需要等待？

系统需要解析文档、清洗文本、分块、生成向量并重建索引。完成后，新文档才能被问答检索到。

### 9.5 为什么回答有参考来源？

因为系统使用 RAG。参考来源表示回答依据了哪些文档片段，能提升可信度，方便用户回看原资料。

### 9.6 语音识别不能用怎么办？

检查麦克风权限、FunASR 依赖和模型下载情况。语音是增强功能，不能用时不影响文字问答。

## 10. 答辩前检查清单

- MySQL 已启动。
- 可以注册和登录。
- API Key 可用。
- 至少准备一份可上传的 PDF、Markdown 或 DOCX 文档。
- 知识文档页面能上传和查看文档。
- 知识问答能生成回答。
- 回答中能看到参考来源。
- 岗位助手能按岗位方向回答。
- 模拟面试能输出追问和分析。
- 学习记录能打开历史会话。
- 知识图谱页面能展示节点。
- 如果要演示语音，提前确认浏览器麦克风权限和 FunASR 可用。
- 如果要校园网演示，提前确认同学能访问 `http://你的IPv4地址:8000`。
