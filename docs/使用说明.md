# 使用说明

这份文档面向项目成员和最终答辩。阅读后应能讲清楚系统做什么、怎么用、整体框架是什么、每个文件负责什么、每个功能板块如何工作，以及项目使用了哪些技术栈。

## 1. 系统概述

大学生求职岗位知识问答系统是一个基于 RAG 的智能问答应用。它把求职资料、岗位说明、面试经验、简历指导等文档导入知识库，经过解析、分块和向量化后用于检索。用户提问时，系统会先从知识库中召回相关片段，再把片段、历史对话和用户问题一起交给大模型生成回答，最后在回答中展示参考来源。

系统适合生产实习答辩展示，因为它包含完整的前端页面、后端接口、数据库、文档处理、向量检索、大模型调用、知识图谱和语音交互流程。

## 2. 使用对象与使用场景

主要用户是准备实习、校招、简历投递、面试和 offer 选择的大学生。

典型场景包括：

- 上传岗位 JD，询问某个岗位需要哪些技能；
- 上传简历指导资料，询问项目经历如何写进简历；
- 进行模拟面试，让 AI 面试官点评并追问；
- 通过知识图谱查看岗位、技能、文档之间的关系；
- 在校园网内让同学一起访问系统进行演示。

## 3. 总体架构

系统采用“前端单页应用 + FastAPI 后端 + MySQL 用户数据 + 本地向量知识库 + 可选 Neo4j 图数据库 + 大模型接口”的架构。

```text
用户浏览器
  │
  │ 访问页面、上传文件、发送问题、语音输入
  ▼
frontend/index.html
  │
  │ fetch / WebSocket
  ▼
FastAPI 后端 api.py
  │
  ├─ user_store.py：用户、令牌、目标管理，连接 MySQL
  ├─ rag_service.py：RAG 问答编排
  ├─ knowledge_base.py：知识库导入、检索、文档管理
  ├─ vector_store.py：Embedding 和向量检索
  ├─ neo4j_store.py：知识图谱和图检索
  ├─ llm_client.py：调用大模型接口
  ├─ conversation.py：历史会话
  └─ speech_service.py：语音识别
```

核心链路：

```text
上传文档
  -> text_processing.py 解析文本
  -> knowledge_base.py 分块和分类
  -> vector_store.py 生成向量并保存索引
  -> neo4j_store.py 可选写入图数据库

用户提问
  -> api.py 接收请求
  -> rag_service.py 组织流程
  -> knowledge_base.py 检索相关片段
  -> llm_client.py 调用大模型
  -> conversation.py 保存会话
  -> 前端流式展示答案和来源
```

## 4. 功能板块说明

### 4.1 登录与个人中心

入口：系统打开后先进入登录/注册界面。

功能：

- 注册账号；
- 用户名、邮箱或手机号登录；
- 修改昵称、邮箱、手机号；
- 上传头像；
- 修改密码；
- 退出登录。

后端实现：

- `api.py` 提供 `/api/auth/register`、`/api/auth/login`、`/api/auth/me`、`/api/auth/avatar`、`/api/auth/password` 等接口。
- `user_store.py` 负责 MySQL 建表、密码哈希、登录令牌和用户资料读写。

答辩讲解点：

- 密码不是明文保存，而是使用 PBKDF2 加盐哈希；
- 登录后前端保存 token，请求接口时放入 Authorization Header；
- 后端根据 token 判断当前用户身份。

### 4.2 首页与目标管理

首页展示系统入口、统计信息和个人求职目标。

功能：

- 查看知识库文档数量、文本块数量等统计；
- 添加求职目标和目标日期；
- 显示距离目标日期的倒计时；
- 删除目标。

后端实现：

- `user_store.py` 中 `user_goals` 表保存目标；
- `api.py` 提供 `/api/auth/goals` 相关接口。

### 4.3 知识问答

这是系统核心功能。用户输入求职问题后，系统返回结构化回答，并展示参考来源。

功能：

- 多轮对话；
- 流式回答；
- 可调召回片段数量；
- 参考来源展示；
- 回答缓存；
- 历史会话保存。

关键流程：

```text
用户问题
  -> RAGService._prepare_request 校验问题
  -> AnswerCache 检查缓存
  -> KnowledgeBase.search 检索知识片段
  -> RAGService._build_messages 构造提示词
  -> LLMClient.stream_chat 调用大模型流式生成
  -> RAGService._append_sources 追加参考来源
  -> ConversationMemory 保存历史记录
```

后端实现：

- `api.py`：`/api/chat` 和 `/api/chat/stream`；
- `rag_service.py`：RAG 主流程和系统提示词；
- `llm_client.py`：大模型接口；
- `conversation.py`：会话历史；
- `cache.py`：回答缓存。

答辩讲解点：

- 系统不是直接把问题发给大模型，而是先检索知识库；
- 检索到的片段会作为上下文传给大模型；
- 回答末尾自动补充“参考来源”，提高可信度；
- 若资料不足，系统提示词要求模型说明不确定，避免编造。

### 4.4 知识文档管理

用于管理系统知识库。

功能：

- 上传 PDF、TXT、Markdown、Word、Excel 等支持格式；
- 自动解析文本；
- 自动分块；
- 自动识别岗位分类；
- 重建向量索引；
- 查看文档列表；
- 按关键词和分类筛选；
- 预览文档内容；
- 删除文档并重建索引。

后端实现：

- `api.py`：`/api/knowledge/upload`、`/api/knowledge/documents`、`/api/knowledge/document/{filename}`、`/api/knowledge/rebuild`；
- `text_processing.py`：读取和清洗不同格式文档；
- `knowledge_base.py`：导入、去重、分类、重建索引、文档列表；
- `vector_store.py`：保存 `chunks.json`、`vectors.npy`、`metadata.json` 和可选 FAISS 索引。

数据位置：

- 内置知识文档：`data/knowledge_base/`
- 用户上传文档：`data/knowledge_base/uploads/`
- 向量索引：`data/index/`

### 4.5 岗位助手

岗位助手面向具体岗位方向进行问答。

功能：

- 展示多个岗位方向；
- 选择岗位后，自动按岗位分类检索相关文档；
- 支持围绕岗位要求、能力准备、学习路线、项目包装继续追问；
- 展示该岗位相关文档。

实现方式：

- 前端在发送问题时传入 `session_type: "job"` 和岗位 `category`；
- `rag_service.py` 根据 `session_type` 改变提示词场景；
- `knowledge_base.py` 按岗位分类过滤召回结果。

### 4.6 模拟面试

模拟面试用于答辩展示 AI 追问能力。

功能：

- AI 扮演面试官；
- 学生输入自我介绍或回答；
- AI 先点评，再提出下一轮追问；
- 支持语音输入；
- 面试记录保存到历史会话。

实现方式：

- 前端页面维护面试消息列表；
- 发送请求时传入 `session_type: "interview"`；
- `rag_service.py` 使用面试场景提示词，让模型按面试官方式回复。

### 4.7 历史记录

功能：

- 查看知识问答、模拟面试、岗位助手三类会话；
- 打开历史会话；
- 重命名会话；
- 删除单个会话；
- 清空会话。

后端实现：

- `api.py`：`/api/sessions`、`/api/history/{session_id}`、`/api/session/{session_id}/rename`、`/api/session/{session_id}/clear`；
- `conversation.py`：负责会话保存、读取、重命名和用户隔离。

### 4.8 知识图谱

知识图谱用于可视化展示岗位、文档、技能等关系。

功能：

- 图谱节点展示；
- 节点详情查看；
- 关键词搜索；
- 岗位分类筛选；
- Neo4j 诊断；
- 支持新增、修改、删除节点和关系的接口。

实现方式：

- `neo4j_store.py` 提取文档中的岗位、技能、公司、城市、学历等实体；
- 写入 Neo4j 后，前端通过图谱接口获取节点和边；
- 如果 Neo4j 未配置或连接失败，`knowledge_base.py` 会返回基于本地文档和分类的兜底图谱。

答辩讲解点：

- 图谱不是系统运行的强依赖；
- Neo4j 可提升关系展示能力；
- 没有 Neo4j 时仍可完成基础演示，体现系统容错设计。

### 4.9 语音能力

功能：

- 录音识别成文字；
- WebSocket 流式识别；
- 浏览器朗读 AI 回答。

后端实现：

- `api.py`：`/api/speech/recognize` 和 `/ws/speech`；
- `speech_service.py`：调用 FunASR 进行识别；
- 前端使用浏览器 `MediaRecorder` 采集语音，使用 `speechSynthesis` 朗读。

## 7. 基本使用流程

1. 启动系统并打开 `http://127.0.0.1:8000`。
2. 注册账号并登录。
3. 进入知识文档页面，上传岗位 JD、面试题、简历指导资料等。
4. 等待系统完成解析和索引重建。
5. 进入知识问答页面提问。
6. 查看回答中的参考来源。
7. 进入岗位助手，选择岗位方向并继续追问。
8. 进入模拟面试，进行自我介绍和追问练习。
9. 进入知识图谱，查看岗位、文档和技能关系。
10. 进入历史记录，复盘或重命名会话。

## 8. 推荐答辩演示流程

建议按以下顺序演示：

1. 介绍项目背景：大学生求职资料分散，查找和准备成本高。
2. 展示系统架构图：前端、后端、MySQL、向量库、Neo4j、大模型。
3. 登录系统，说明用户数据和会话隔离。
4. 上传一份岗位 JD 或面试资料，说明解析、分块、向量化和索引重建。
5. 在知识问答中提问，展示流式回答和参考来源。
6. 切换岗位助手，展示按岗位分类检索。
7. 切换模拟面试，展示 AI 点评和追问。
8. 切换知识图谱，展示岗位、文档、技能关系。
9. 展示历史记录和目标管理。
10. 总结技术亮点：RAG、流式输出、知识库、图谱兜底、语音输入、MySQL 用户体系。

## 9. 常见问题

### 为什么回答里有参考来源？

因为系统使用 RAG。回答不是只依赖模型记忆，而是优先依据知识库召回片段生成，所以需要展示来源让用户判断可信度。

### 没有 Neo4j 能不能运行？

可以。Neo4j 是增强能力。未配置时，系统会使用本地知识库生成基础图谱，问答和文档管理仍可使用。

### Hugging Face 模型没下载怎么办？

系统会自动降级到本地 hashing embedding 检索，适合课堂演示。若要提升检索效果，按部署手册下载 sentence-transformers 模型。

### 为什么上传文档后要重建索引？

因为新文档需要解析成文本块，再生成向量并写入索引。只有重建后，用户提问时才能检索到新资料。

### 答辩时最应该强调什么？

重点强调完整链路：文档入库、向量检索、提示词增强、大模型回答、来源追溯、历史保存和图谱展示。这比单纯展示聊天更能体现项目完整度。
