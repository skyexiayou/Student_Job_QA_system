# 使用说明

本文档基于当前项目代码整理，面向项目组成员、演示人员和最终答辩使用。阅读后应能够说明系统的整体框架、功能板块、主要文件作用、数据流转过程、技术栈和答辩展示路线。

## 1. 系统简介

大学生求职岗位知识问答系统是一个面向实习、校招、简历、面试和岗位选择场景的 RAG 智能问答系统。

系统允许用户上传求职相关资料，例如岗位 JD、企业招聘资料、面试题、简历指导文档和行业信息。后端会对文档进行解析、清洗、分块、向量化和索引构建。用户提问时，系统先从知识库中检索相关内容，再结合历史对话和检索片段调用大模型生成回答，并在回答中展示参考来源。

项目不仅是一个聊天页面，而是包含用户体系、知识库管理、岗位助手、模拟面试、知识图谱、语音输入、历史记录和目标管理的完整应用。

## 2. 用户与使用场景

目标用户是正在准备实习和校园招聘的大学生。

典型使用场景：

- 想了解前端、后端、算法、测试、产品等岗位的能力要求；
- 想把自己的课程设计、竞赛项目或实习经历写进简历；
- 想根据岗位 JD 准备技能清单和学习路线；
- 想进行模拟面试，让 AI 面试官点评并追问；
- 想上传企业资料或面试资料，快速问答和定位重点；
- 想在项目答辩中展示一个完整的 AI 应用系统。

## 3. 系统整体架构

系统采用前后端一体部署方式：FastAPI 后端负责接口和静态资源服务，前端是 Vue 单页应用，浏览器直接访问后端根路径即可进入系统。

```text
浏览器用户
  |
  | 访问页面、登录注册、上传文档、发送问题、录音输入
  v
frontend/index.html
  |
  | fetch HTTP 请求 / WebSocket 语音连接
  v
FastAPI 后端 src/rag_job_qa/api.py
  |
  +-- user_store.py        用户、登录令牌、头像、目标管理，连接 MySQL
  +-- knowledge_base.py    知识库导入、分类、检索、文档管理、图谱兜底
  +-- vector_store.py      Embedding、FAISS/numpy 检索、索引持久化
  +-- rag_service.py       RAG 主流程，构造提示词，调用大模型，追加来源
  +-- llm_client.py        大模型普通和流式接口调用
  +-- conversation.py      多轮会话和历史记录
  +-- neo4j_store.py       可选 Neo4j 知识图谱
  +-- speech_service.py    FunASR 语音识别
  +-- cache.py             问答缓存
  +-- models.py            数据模型
```

核心业务链路分为两条。

文档入库链路：

```text
上传 PDF/TXT/Markdown/DOCX/XLSX
  -> text_processing.py 解析文本
  -> clean_text 清洗文本
  -> split_text 分块
  -> knowledge_base.py 判断岗位分类并导入
  -> vector_store.py 生成向量
  -> data/index/ 保存 chunks、vectors、metadata
  -> neo4j_store.py 可选写入 Neo4j 图谱
```

问答生成链路：

```text
用户提问
  -> api.py 接收 /api/chat/stream
  -> rag_service.py 检查问题、会话和缓存
  -> knowledge_base.py 检索相关知识片段
  -> rag_service.py 构造系统提示词和用户提示词
  -> llm_client.py 调用大模型流式生成
  -> rag_service.py 追加参考来源
  -> conversation.py 保存历史记录
  -> 前端逐字显示回答
```

## 4. 前端页面与功能板块

当前前端集中在 `frontend/index.html`，使用 Vue 3 单页应用实现。页面通过 hash 路由切换，主要页面包括：

| --- | --- | --- |
| 首页 | `#home` | 展示系统入口、热门岗位、模拟面试入口、知识文档入口、求职目标倒计时。 |
| 岗位助手 | `#jobs` | 展示岗位方向，按岗位分类关联文档，围绕岗位进行问答。 |
| 知识图谱 | `#graph` | 展示岗位、文档、技能等节点和关系，支持搜索和分类筛选。 |
| 知识问答 | `#chat` | RAG 多轮问答，支持流式回答、语音输入、来源展示和朗读。 |
| 模拟面试 | `#interview` | AI 面试官追问，输出“面试官回应”和“简短分析”。 |
| 知识文档 | `#docs` | 上传、查看、筛选、预览、删除知识文档。 |
| 学习记录 | `#history` | 查看知识问答、岗位助手、模拟面试历史，支持重命名和删除。 |

### 4.1 登录注册与个人中心

系统打开后，如果本地没有 token，会进入登录/注册界面。

支持功能：

- 用户注册；
- 用户名、邮箱或手机号登录；
- 退出登录；
- 修改昵称、邮箱、手机号；
- 上传头像；
- 修改密码。

相关接口：

```text
POST /api/auth/register
POST /api/auth/login
GET  /api/auth/me
POST /api/auth/logout
PUT  /api/auth/me
POST /api/auth/avatar
PUT  /api/auth/password
```

实现文件：

- `api.py`：定义认证接口；
- `user_store.py`：连接 MySQL，创建用户表和令牌表；
- `frontend/index.html`：保存 token 到 localStorage，并在请求时带上 Authorization Header。

答辩要点：

- 用户密码使用 PBKDF2 加盐哈希保存，不保存明文密码；
- 登录成功后生成 token；
- 后端通过 token 判断用户身份，实现用户数据隔离。

### 4.3 知识问答

知识问答是系统核心。用户输入问题后，系统会基于知识库检索结果生成回答。

支持功能：

- 多轮问答；
- 流式回答；
- 参考来源展示；
- 可设置召回片段数量；
- 语音输入；
- 回答朗读；
- 回答复制；
- 停止生成；
- 历史会话保存。

相关接口：

```text
POST /api/chat
POST /api/chat/stream
GET  /api/sessions
GET  /api/history/{session_id}
POST /api/session/new
POST /api/session/{session_id}/clear
POST /api/session/{session_id}/rename
DELETE /api/sessions
```

实现文件：

- `rag_service.py`：RAG 编排、系统提示词、场景提示词、来源追加；
- `knowledge_base.py`：检索知识片段；
- `vector_store.py`：向量相似度搜索；
- `llm_client.py`：调用大模型；
- `conversation.py`：保存历史对话；
- `cache.py`：缓存重复问题。

答辩要点：

- 系统并不是直接问大模型，而是先查知识库；
- RAG 可以减少模型凭空回答的问题；
- 回答末尾有参考来源，便于追溯；
- 流式输出使用 `application/x-ndjson`，前端逐行读取事件。

### 4.5 岗位助手

岗位助手围绕具体岗位方向进行问答。前端内置多个岗位方向，后端使用岗位分类过滤知识片段。

当前后端岗位分类主要包括：

```text
backend     后端开发
frontend    前端开发
client      客户端开发
embedded    嵌入式底层开发
ai          AI 算法
bigdata     大数据开发
cloud       云计算云原生
security    网络安全
testing     测试体系
game        游戏开发
ic          芯片 IC 开发
nondev      技术非开发岗
```

使用方式：

1. 进入岗位助手页面；
2. 选择一个岗位方向；
3. 查看该岗位相关文档；
4. 输入问题，例如“这个岗位需要准备哪些项目？”；
5. 系统带着岗位分类进行检索和回答。

实现方式：

- 前端发送 `session_type: "job"`；
- 前端发送岗位 `category`；
- `knowledge_base.py` 根据分类过滤召回结果；
- `rag_service.py` 为岗位助手加入岗位建议、能力要求、学习路径等场景提示。

### 4.6 模拟面试

模拟面试页面让 AI 扮演面试官。

支持功能：

- 自动生成开场；
- 用户输入自我介绍或回答；
- AI 输出“面试官回应”；
- AI 输出“简短分析”；
- AI 继续提出一个具体追问；
- 支持语音输入；
- 支持停止生成；
- 面试会话进入历史记录。

实现方式：

- 前端发送 `session_type: "interview"`；
- 前端将候选人回答包装成面试官提示词；
- `rag_service.py` 使用面试场景提示；
- 回答仍会结合知识库检索结果。

答辩展示建议：

可以先让同学说一段简短自我介绍，再展示 AI 如何追问，例如让 AI 追问项目细节、技术难点或岗位匹配度。

知识图谱页面展示岗位、文档、技能等节点关系。

支持功能：

- 按岗位分类筛选；
- 搜索节点；
- 点击节点查看详情；
- 查看节点关联关系；
- Neo4j 连接诊断；
- 后端提供节点和关系的增删改接口。

相关接口：

```text
GET    /api/graph/categories
GET    /api/graph/diagnostics
GET    /api/graph
GET    /api/graph/node/{node_id}
GET    /api/graph/search
POST   /api/graph/node
PUT    /api/graph/node/{node_id}
DELETE /api/graph/node/{node_id}
POST   /api/graph/relation

```

实现文件：

- `neo4j_store.py`：连接 Neo4j、抽取实体、构建图谱、图检索；
- `knowledge_base.py`：当 Neo4j 不可用时返回本地兜底图谱；
- `frontend/index.html`：使用图谱画布展示节点关系。

答辩要点：

- Neo4j 是增强模块，不是系统强依赖；
- 未配置 Neo4j 时，系统仍会展示岗位分类和文档关系；
- 这样能保证答辩时即使图数据库没启动，也能完成基础演示。

### 4.8 语音输入与朗读

系统支持语音输入和浏览器朗读。

支持功能：

- 知识问答语音输入；
- 模拟面试语音输入；
- WebSocket 流式语音识别；
- 普通音频上传识别；
- 浏览器 `speechSynthesis` 朗读回答。

相关接口：

```text
POST      /api/speech/recognize
WebSocket /ws/speech
```

实现文件：

- `speech_service.py`：封装 FunASR；
- `api.py`：提供语音识别 HTTP 和 WebSocket 接口；
- `frontend/index.html`：调用浏览器 `MediaRecorder` 录音。

注意：

- 首次加载 FunASR 模型可能较慢；
- 浏览器录音需要允许麦克风权限；
- 如果 FunASR 依赖未安装，语音功能不可用，但其他功能不受影响。

## 7. 配置项说明

主要配置来自 `.env`、系统环境变量和 API Key CSV。

常用配置：

| 配置 | 作用 | 默认值 |
| --- | --- | --- |
| `QWEN_API_KEY` / `DASHSCOPE_API_KEY` / `OPENAI_API_KEY` | 大模型 API Key | 空 |
| `QWEN_BASE_URL` / `DASHSCOPE_BASE_URL` / `OPENAI_BASE_URL` | OpenAI 兼容接口地址 | 空 |
| `QWEN_MODEL` / `OPENAI_MODEL` | 大模型名称 | `qwen-plus` |
| `LLM_TIMEOUT` | 大模型请求超时 | `60` |
| `LLM_MAX_TOKENS` | 大模型最大输出 token | `650` |
| `EMBEDDING_MODEL` | Embedding 模型 | `sentence-transformers/all-MiniLM-L6-v2` |
| `USE_HF_EMBEDDING` | 是否使用 Hugging Face Embedding | `true` |
| `HF_LOCAL_FILES_ONLY` | 是否只从本地加载 HF 模型 | `true` |
| `CHUNK_SIZE` | 文本分块大小 | `650` |
| `CHUNK_OVERLAP` | 分块重叠长度 | `100` |
| `DEFAULT_TOP_K` | 默认召回片段数 | `4` |
| `MAX_UPLOAD_SIZE_MB` | 单文件上传大小上限 | `30` |
| `MEMORY_ROUNDS` | 历史对话保留轮数 | `4` |
| `CACHE_SIZE` | 回答缓存容量 | `128` |
| `MYSQL_HOST` | MySQL 地址 | `127.0.0.1` |
| `MYSQL_PORT` | MySQL 端口 | `3306` |

## 8. 基本使用流程

1. 启动后端服务。

3. 注册或登录账号。

## 9. 答辩推荐讲解顺序

建议答辩时按照“问题背景 -> 系统架构 -> 核心流程 -> 功能演示 -> 技术亮点 -> 总结”的顺序讲。

### 9.1 问题背景

大学生求职资料来源分散，岗位 JD、面试题、简历经验和行业信息往往分布在不同文件中。普通搜索无法结合个人问题和已上传资料生成结构化建议。本系统通过 RAG 把本地知识库和大模型结合起来，提升求职资料利用效率。

### 9.2 系统架构

重点讲：

- 前端 Vue 单页应用；
- 后端 FastAPI；
- MySQL 用户体系；
- 本地向量知识库；
- 大模型接口；
- 可选 Neo4j 知识图谱；
- 可选 FunASR 语音识别。

### 9.3 核心流程

重点讲两条链路：

- 文档上传后如何解析、分块、向量化和索引；
- 用户提问后如何检索、构造提示词、调用大模型和展示来源。

### 9.4 演示流程

推荐现场演示：

1. 登录系统；
2. 展示首页和目标管理；
3. 上传一份岗位 JD 或面试资料；
4. 在知识问答中提问；
5. 展示参考来源；
6. 切换岗位助手，按岗位继续追问；
7. 切换模拟面试，让 AI 追问；
8. 展示知识图谱；
9. 展示学习记录；
10. 总结技术亮点。

### 9.5 技术亮点

可以强调：

- RAG 检索增强生成；
- 流式回答；
- 支持多格式文档解析；
- 向量检索支持 FAISS，也支持 numpy 降级；
- Hugging Face 模型不可用时可降级到 hashing embedding；
- Neo4j 不可用时可降级到本地图谱；
- MySQL 用户体系和 token 登录；
- 语音输入和朗读提升交互体验。

## 10. 常见问题

### 10.1 没有 API Key 能不能运行？

页面、登录、知识库部分可以检查，但大模型回答会受影响。答辩前应确认 API Key 可用。

### 10.2 没有 Neo4j 能不能运行？

可以。Neo4j 是增强模块，不是必需模块。没有 Neo4j 时，系统仍会使用本地知识库生成基础图谱。

### 10.3 Hugging Face 模型没有下载怎么办？

系统会降级到本地 hashing embedding，可以保证演示可运行。检索效果可能不如正式 Embedding 模型。

### 10.4 上传文档后为什么需要等待？

系统需要解析文档、清洗文本、分块、生成向量并重建索引。完成后，新文档才能被问答检索到。

### 10.5 为什么回答有参考来源？

因为系统使用 RAG。参考来源表示回答依据了哪些文档片段，能提升可信度，方便用户回看原资料。

### 10.6 语音识别不能用怎么办？

检查麦克风权限、FunASR 依赖和模型下载情况。语音是增强功能，不能用时不影响文字问答。
