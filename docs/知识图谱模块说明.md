# 知识图谱模块说明

本文档整理本项目中 Neo4j 知识图谱相关的设计、配置、实体抽取、关系构建、查询接口和前端可视化数据结构。对应代码主要在：

- `src/rag_job_qa/neo4j_store.py`：Neo4j 连接、建图、查询、节点/关系 CRUD。
- `src/rag_job_qa/knowledge_base.py`：知识库重建、文件导入、向量检索和图谱检索入口。
- `src/rag_job_qa/api.py`：FastAPI 图谱接口。
- `frontend/index.html`：知识图谱页面和 ECharts 可视化。

## 1. 模块职责

知识图谱模块在系统里的职责是把“知识库文档”转换成“可查询、可展示、可辅助问答的图结构”。

整体流程如下：

1. 用户上传 PDF、Word、Markdown、文本等知识文档。
2. `KnowledgeBase.rebuild()` 解析全部文档，调用 `build_chunks()` 切分为多个 `DocumentChunk`。
3. 每个 chunk 根据内容被归入一个岗位方向，例如 `backend`、`frontend`、`ai`。
4. 向量库生成每个 chunk 的 embedding。
5. `Neo4jKnowledgeStore.rebuild()` 将文档、切片、岗位方向、岗位、技能、城市、学历、薪资等写入 Neo4j。
6. QA 检索时优先走 Neo4j 向量索引，并通过图关系补充相关上下文。
7. 前端通过 `/api/graph`、`/api/graph/node/{node_id}`、`/api/graph/search` 获取图谱数据并用 ECharts 展示。

简单理解：向量库负责“语义相似”，Neo4j 负责“实体之间的显式关系”。

## 3. 核心概念

### 3.1 节点、关系、属性

Neo4j 是图数据库，数据由三部分组成：

- 节点 Node：表示一个对象，例如文档、岗位方向、技能。
- 关系 Relationship：表示两个节点之间的连接，例如“文档属于某方向”“岗位需要某技能”。
- 属性 Properties：节点或关系上的键值对，例如 `name`、`title`、`source`。

截图里的 `TYPE: Document`，对应的就是节点类型。代码里这个值来自：

```cypher
labels(n)[0] AS type
```

也就是 Neo4j 节点的第一个 Label。对于 Document 节点，Label 是 `Document`，同时属性里也会设置 `type = 'Document'`。

### 3.2 一跳关联

“一跳关联”指从当前节点出发，只沿着一条关系边能到达的邻居节点。

例如：

```text
(Document)-[:RELATED_TO]->(JobCategory)
```

从 `Document` 到 `JobCategory` 只经过一条边，所以是 1-hop，也叫一跳。

节点详情接口 `/api/graph/node/{node_id}` 查询的就是一跳关系：

```cypher
MATCH (n)
WHERE elementId(n) = $node_id
OPTIONAL MATCH (n)-[r]-(m)
```

这里 `(n)-[r]-(m)` 不限定方向，所以入边和出边都会查出来。

### 3.3 二跳关联

“二跳关联”指最多经过两条关系边能到达的节点。

代码中图谱分类筛选和搜索会用到 0 到 2 跳寻找分类：

```cypher
OPTIONAL MATCH (n)-[*0..2]-(jc:JobCategory)
```

含义是：从节点 `n` 出发，经过 0、1、2 条边，看能不能找到 `JobCategory`。这样即使一个节点没有直接连到分类，也可以通过文档、岗位等中间节点归到某个分类下。

## 4. 节点类型

### 4.1 自动建图涉及的节点

| Neo4j Label / type | 含义 | 主要属性 | 来源 |
| --- | --- | --- | --- |
| `JobCategory` | 岗位大类/方向 | `key`、`name`、`work`、`skills`、`type` | 系统内置 12 类岗位方向。 |
| `Document` | 文档 | `source`、`title`、`name`、`type` | 每个知识文件合并为一个文档节点。 |
| `Chunk` | 文档切片 | `chunk_id`、`source`、`title`、`content`、`metadata_json`、`embedding` | 文档被切分后的片段。 |
| `Position` | 岗位/职位 | `name`、`type` | 从 chunk 内容中抽取。 |
| `Company` | 公司 | `name`、`type` | 从 chunk 内容中抽取。 |
| `Skill` | 技能 | `name`、`type` | 从 chunk 内容中抽取。 |
| `Salary` | 薪资 | `name`、`type` | 从 chunk 内容中抽取。 |
| `City` | 城市 | `name`、`type` | 从 chunk 内容中抽取。 |
| `Education` | 学历要求 | `name`、`type` | 从 chunk 内容中抽取。 |

### 4.2 CRUD 允许的节点类型

手动新增节点时只允许以下 Label：

```text
JobCategory
Position
Company
Skill
Salary
City
Education
Document
```

注意：`Chunk` 会在知识库重建时自动创建，不在手动 CRUD 白名单里。

### 4.3 前端实际展示的节点类型

`/api/graph` 主图谱接口只返回以下节点：

```text
JobCategory
Position
Skill
Document
```

这是为了让主图更清晰，避免把所有 chunk、城市、薪资、学历都铺开导致图谱过密。

前端 ECharts 颜色分类包含：

```text
JobCategory
Position
Skill
Document
Chunk
```

其中 `Chunk` 是兜底分类：如果后端返回未知类型，前端会按 `Chunk` 样式处理。

## 5. 岗位分类

图谱内置 12 个岗位方向，`key` 是接口和筛选使用的稳定值。

| key | 含义 |
| --- | --- |
| `backend` | 后端开发 |
| `frontend` | 前端开发 |
| `client` | 客户端开发 |
| `embedded` | 嵌入式/底层开发 |
| `ai` | AI 算法 |
| `bigdata` | 大数据开发 |
| `cloud` | 云计算/云原生/运维 |
| `security` | 网络安全 |
| `testing` | 测试/质量保障 |
| `game` | 游戏开发 |
| `ic` | 芯片/IC/FPGA |
| `nondev` | 技术非开发岗，例如产品、售前、技术支持 |

分类逻辑在 `classify_job_category(text)` 中，通过关键词命中打分决定。没有命中时默认归为 `nondev`。

## 6. 实体抽取

实体抽取由 `JobEntityExtractor.extract(text)` 完成。它不是大模型抽取，而是轻量规则抽取，适合实习项目演示和本地快速运行。

### 6.1 抽取结果结构

内部实体对象是：

```python
@dataclass(frozen=True)
class JobEntity:
    kind: str
    name: str
```

`kind` 决定写入哪个 Neo4j Label，`name` 是实体名称。

### 6.2 kind 到 Label 的映射

```text
position  -> Position
company   -> Company
skill     -> Skill
salary    -> Salary
city      -> City
education -> Education
```

### 6.3 抽取规则

| kind | 抽取方式 | 例子 |
| --- | --- | --- |
| `skill` | 命中预置技能词表 | `Python`、`Java`、`Vue`、`React`、`Docker`、`Redis`、`PyTorch`、`Spark`、`Verilog` |
| `education` | 命中预置学历词表 | 大专、本科、硕士、博士、不限学历等 |
| `city` | 命中预置城市词表 | 北京、上海、深圳、西安等 |
| `company` | 正则匹配公司后缀 | 包含“公司、集团、科技、网络、信息、有限公司”等后缀的名称 |
| `position` | 正则匹配职位后缀 | 工程师、开发、产品经理、测试、算法、运维、设计师、实习生等 |
| `salary` | 正则匹配薪资表达 | `10-20K`、`15k`、年薪表达、薪资面议等 |

## 7. 关系类型

### 7.1 系统允许手动创建的关系类型

```text
RELATED_TO
CONTAINS
MENTIONS
REQUIRES
OFFERS
LOCATED_IN
REQUIRES_EDUCATION
```

新增关系时，`type` 会被转成大写并校验，不在白名单内会报：

```text
Unsupported relation type: xxx
```

### 7.2 自动建图关系

| 关系 | 含义 | 自动建图场景 |
| --- | --- | --- |
| `(Document)-[:RELATED_TO]->(JobCategory)` | 文档属于某岗位方向 | 每个文档根据分类结果连接到岗位方向。 |
| `(Document)-[:HAS_CHUNK]->(Chunk)` | 文档包含切片 | 每个文档切片创建后连接回文档。 |
| `(Chunk)-[:RELATED_TO]->(JobCategory)` | 切片属于某岗位方向 | 每个 chunk 根据分类结果连接到岗位方向。 |
| `(Chunk)-[:MENTIONS]->(Position)` | 切片提到岗位 | 抽取到 `position` 实体时创建。 |
| `(JobCategory)-[:CONTAINS]->(Position)` | 岗位方向包含岗位 | 抽取到岗位后，方向连接岗位。 |
| `(Document)-[:RELATED_TO]->(Position)` | 文档关联岗位 | 抽取到岗位后，文档连接岗位。 |
| `(Chunk)-[:MENTIONS]->(Company)` | 切片提到公司 | 抽取到 `company` 实体时创建。 |
| `(Chunk)-[:REQUIRES]->(Skill)` | 切片要求技能 | 抽取到 `skill` 实体时创建。 |
| `(JobCategory)-[:REQUIRES]->(Skill)` | 岗位方向需要技能 | 抽取到技能后，方向连接技能。 |
| `(Position)-[:REQUIRES]->(Skill)` | 岗位需要技能 | 当前方向下已有岗位时，把岗位和技能连接。 |
| `(Chunk)-[:OFFERS]->(Salary)` | 切片提供薪资信息 | 抽取到 `salary` 实体时创建。 |
| `(Chunk)-[:LOCATED_IN]->(City)` | 切片关联城市 | 抽取到 `city` 实体时创建。 |
| `(Chunk)-[:REQUIRES_EDUCATION]->(Education)` | 切片要求学历 | 抽取到 `education` 实体时创建。 |

## 8. 建图流程

### 8.1 触发入口

以下操作会触发知识库重建，从而同步重建 Neo4j 图谱：

- `POST /api/knowledge/rebuild`
- `POST /api/knowledge/upload`
- 删除文档后重新加载图谱时，取决于知识库是否重建

核心调用链：

```text
api.py
  -> KnowledgeBase.rebuild()
    -> build_chunks()
    -> VectorStore.rebuild()
    -> Neo4jKnowledgeStore.rebuild()
      -> _ensure_vector_index()
      -> _replace_graph()
```

### 8.2 重建时删除什么

`_replace_graph()` 会先删除：

```cypher
MATCH (c:Chunk) DETACH DELETE c
MATCH (d:Document) DETACH DELETE d
```

然后删除无关系的实体节点：

```cypher
MATCH (e:Position) WHERE NOT (e)--() DETACH DELETE e
...
```

这意味着知识库文档和切片是按当前知识库全量重建的。手动创建的孤立实体节点如果没有任何关系，可能在后续重建时被清理。

### 8.3 向量索引

Neo4j 中为 `Chunk.embedding` 创建向量索引：

```cypher
CREATE VECTOR INDEX job_chunk_embedding_index IF NOT EXISTS
FOR (c:Chunk) ON (c.embedding)
OPTIONS {
  indexConfig: {
    `vector.dimensions`: 384,
    `vector.similarity_function`: 'cosine'
  }
}
```

索引名称由 `NEO4J_VECTOR_INDEX` 控制。代码会校验索引名格式，只允许字母、数字、下划线，并且不能以数字开头；非法时退回 `job_chunk_embedding_index`。

## 9. 检索增强逻辑

问答检索入口在 `KnowledgeBase.search()`。

流程：

1. 对用户问题生成 query embedding。
2. 调用 `graph_store.search()`，优先使用 Neo4j 向量索引查 `Chunk`。
3. 如果 Neo4j 有结果，则调用 `graph_store.enrich()` 补充图谱实体信息。
4. 如果 Neo4j 不可用或没有结果，则退回本地向量库检索。

### 9.1 Neo4j 向量检索

调用：

```cypher
CALL db.index.vector.queryNodes($index_name, $top_k, $embedding)
YIELD node, score
```

返回字段：

| 字段 | 含义 |
| --- | --- |
| `chunk_id` | 命中的 chunk id |
| `source` | 来源文件 |
| `title` | 标题 |
| `content` | 原 chunk 内容，可能拼接相关图上下文 |
| `metadata_json` | chunk 元数据 JSON |
| `score` | Neo4j 向量相似度分数 |

### 9.2 图上下文补充

命中某个 chunk 后，会查找和它共享实体的其它 chunk：

```cypher
OPTIONAL MATCH (node)-[]-(entity)<-[]-(neighbor:Chunk)
WHERE neighbor.chunk_id <> node.chunk_id
```

含义是：

```text
当前 chunk -> 某个实体 <- 其它 chunk
```

如果找到了其它相关 chunk，会最多取 2 条内容追加到 `[Related graph context]` 中。

### 9.3 来源信息增强

`enrich()` 会把命中 chunk 的一跳实体整理到 metadata：

```text
graph_entities = "Skill:Python; Position:后端开发工程师; ..."
```

前端聊天来源中会显示：

```text
retrieval_mode = neo4j_vector_graph
graph_entities = ...
```

如果退回本地向量库，则：

```text
retrieval_mode = local_vector_fallback
```

模糊搜索图谱节点。

Query 参数：

| 参数 | 默认值 | 限制 | 含义 |
| --- | --- | --- | --- |
| `keyword` | 必填 | 非空字符串 | 搜索关键词。 |
| `category` | 空字符串 | 可为空或 12 类 key | 限定岗位方向。 |
| `limit` | `30` | 后端限制到 1 到 100 | 返回数量。 |

搜索范围：

```text
JobCategory
Position
Skill
Document
```

匹配字段：

```text
name/title/source/key
```

返回 data：

```json
{
  "items": [
    {
      "id": "4:xxxx:12",
      "label": "Java 后端开发",
      "type": "Position",
      "properties": {
        "name": "Java 后端开发",
        "type": "Position"
      }
    }
  ]
}
```

### 10.6 POST /api/graph/node

新增或合并节点。

请求体：

```json
{
  "label": "Skill",
  "properties": {
    "name": "FastAPI",
    "description": "Python Web API 框架"
  }
}
```

参数说明：

| 参数 | 必填 | 含义 |
| --- | --- | --- |
| `label` | 是 | 节点 Label，只能是 CRUD 白名单中的类型。 |
| `properties` | 否 | 节点属性对象。必须能从 `name/title/key` 中得到一个非空名称。 |

身份字段规则：

- `JobCategory` 且传了 `key` 时，按 `key` 做 `MERGE`。
- 其它节点按 `name` 做 `MERGE`。
- 后端会自动设置 `properties.type = label`。

### 10.7 PUT /api/graph/node/{node_id}

更新节点属性。

请求体：

```json
{
  "properties": {
    "name": "FastAPI",
    "level": "important"
  }
}
```

说明：后端执行 `SET n += $properties`，也就是增量覆盖；没有传的旧属性不会被删除。

### 10.8 DELETE /api/graph/node/{node_id}

删除节点。

后端执行：

```cypher
DETACH DELETE n
```

也就是会连同该节点上的所有关系一起删除。

### 10.9 POST /api/graph/relation

新增或合并关系。

请求体：

```json
{
  "source_id": "4:xxxx:1",
  "target_id": "4:xxxx:2",
  "type": "REQUIRES",
  "properties": {
    "source": "manual"
  }
}
```

参数说明：

| 参数 | 必填 | 含义 |
| --- | --- | --- |
| `source_id` | 是 | 起点节点 `elementId`。 |
| `target_id` | 是 | 终点节点 `elementId`。 |
| `type` | 是 | 关系类型，只能是关系白名单。 |
| `properties` | 否 | 关系属性。 |

后端执行：

```cypher
MERGE (source)-[r:TYPE]->(target)
SET r += $properties
```

### 10.10 DELETE /api/graph/relation/{relation_id}

删除关系。

`relation_id` 是 Neo4j `elementId(relationship)`。

## 11. 前端可视化

知识图谱页面在 `frontend/index.html`，路由 hash 是：

```text
#graph
```

### 11.1 页面结构

页面分为三栏：

- 左侧：关键词搜索、岗位分类筛选、节点列表。
- 中间：ECharts graph 画布。
- 右侧：选中节点详情、一跳关系列表。

### 11.2 数据加载

进入图谱页时执行：

```text
loadGraph()
```

它会并行请求：

```text
GET /api/graph/categories
GET /api/graph?category=<graphCategory>&limit=180
```

### 11.3 ECharts 节点分类颜色

前端按 `node.type` 分类：

```text
JobCategory
Position
Skill
Document
Chunk
```

也就是图谱底部图例中的这几类：

| 图例名称 | 含义 | 在图谱中的作用 |
| --- | --- | --- |
| `JobCategory` | 岗位大类 | 表示系统内置的 12 个岗位方向，例如后端、前端、AI、大数据等。 |
| `Position` | 岗位/职位 | 表示从文档内容里抽取出来的具体岗位，例如后端开发、产品经理、测试工程师。 |
| `Skill` | 技能 | 表示岗位或文档中提到的能力要求，例如 Java、Python、Vue、Docker、SQL。 |
| `Document` | 文档 | 表示上传或内置知识库中的一个完整文件，例如某个 PDF、Markdown、简历或政策文档。 |
| `Chunk` | 文档切片 | 表示文档被切分后的内容片段，主要用于向量检索和图谱增强；主图接口通常不直接展示大量 Chunk，前端保留它作为兜底类型和扩展展示类型。 |

节点大小：

| type | symbolSize |
| --- | --- |
| `JobCategory` | 56 |
| `Document` | 38 |
| `Position` | 30 |
| 其它 | 22 |

## 12. 常用 Cypher 排查语句

### 12.1 查看所有节点类型数量

```cypher
MATCH (n)
RETURN labels(n)[0] AS type, count(*) AS count
ORDER BY count DESC;
```

### 12.2 查看所有关系类型数量

```cypher
MATCH ()-[r]->()
RETURN type(r) AS type, count(*) AS count
ORDER BY count DESC;
```

### 12.3 查看某个文档的一跳关系

```cypher
MATCH (d:Document {source: "文件名.pdf"})-[r]-(m)
RETURN d, r, m;
```

### 12.4 查看岗位方向和技能

```cypher
MATCH (jc:JobCategory)-[r:REQUIRES]->(s:Skill)
RETURN jc.key, jc.name, type(r), s.name
ORDER BY jc.key, s.name;
```

### 12.5 查看向量索引

```cypher
SHOW INDEXES
YIELD name, type, labelsOrTypes, properties
WHERE name = "job_chunk_embedding_index"
RETURN *;
```

## 13. 常见问题

### 13.1 Neo4j 未连接

表现：

```json
{
  "nodes": [],
  "edges": [],
  "neo4j_enabled": false
}
```

排查：

1. 检查 `.env` 是否配置了 `NEO4J_URI` 和 `NEO4J_PASSWORD`。
2. 打开 `GET /api/graph/diagnostics` 看 `socket_open` 和 `driver_ok`。
3. 如果 `socket_open = false`，说明 Bolt 端口没启动。
4. 如果 `socket_open = true` 但 `driver_ok = false`，通常是账号、密码或数据库名错误。

### 13.2 图谱里只有 Document 和 JobCategory

原因通常是实体抽取规则没有命中岗位、技能、公司、城市等关键词。

可以检查：

- 文档解析出来的文本是否正常。
- 文档中是否包含技能词表或职位后缀。
- `JobEntityExtractor.skill_words`、`education_words`、`city_words` 是否覆盖当前数据。

### 13.3 前端只显示部分节点

`/api/graph` 接口会限制数量：

- 参数 `limit` 默认 240。
- 后端限制范围 20 到 500。
- 前端当前传 180。

另外，主图只返回 `JobCategory`、`Position`、`Skill`、`Document`，不会展示所有 `Chunk`、`Company`、`Salary`、`City`、`Education`。

### 13.4 elementId 是什么

`elementId()` 是 Neo4j 给节点或关系的内部元素 id。前端用它来做详情查询、更新和删除。

注意：`elementId` 不适合当长期业务 id。数据库重建、导入导出后可能变化。本项目把它作为前端当前会话里的操作 id 使用。
